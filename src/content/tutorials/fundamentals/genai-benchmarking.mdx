This tutorial is a **general benchmarking guide** for running **LLMs** and **VLMs** on Jetson with **vLLM**. The goal is to help you measure performance in a simple, repeatable way, then understand what the numbers actually mean.

We will use one model as a concrete example, but the workflow applies more broadly. The two things that matter most are:

- keep the test conditions consistent
- understand what each metric is really telling you

## What This Guide Measures

This guide is about **performance**, not answer quality. We are trying to answer questions like:

- How long does a request take to start responding?
- How smooth is token streaming once generation starts?
- How much total work can the system handle under load?

We are **not** measuring whether the model is correct, helpful, or better than another model at reasoning.

## The Main Metrics

### Time to First Token (TTFT)

TTFT is how long it takes before the first output token appears. This includes the initial prompt processing phase, so it is one of the most important numbers for responsiveness.

### Output Token Throughput (tok/s)

This is the total number of **generated output tokens** divided by the total benchmark time. Keep it simple:

```text
output token throughput = total generated tokens / total elapsed time
```

This is **not** just raw generation speed after the first token. It already includes the time spent before generation starts, including **TTFT**, along with the rest of the request lifecycle. That is why this number is useful: it reflects the real end-to-end serving behavior more than a pure decode-only metric would.

### Inter-Token Latency (ITL)

ITL is the average delay between generated tokens after the first one. Lower ITL usually means the stream feels smoother.

### End-to-End Latency (E2EL)

E2EL is the total time for a request from start to finish. If you want the simplest "how long did this request take?" number, this is it.

## How to Run a Useful Benchmark

Before running commands, keep these rules in mind:

- Use the **same model**, **same prompt length**, **same output length**, and **same concurrency** when comparing runs.
- Always do a **warm-up** run first.
- For **VLMs**, benchmark with a dataset that actually contains images.
- If you change several parameters at once, the comparison stops being clean.

## 2. The Benchmarking Workflow

The easiest workflow uses **two terminals**:

- **Terminal 1** serves the model
- **Terminal 2** sends benchmark traffic to that running server

In **Terminal 2**, open a shell in the same running container:

```bash
sudo docker exec -it vllm bash
```

You should now have two terminals inside the same container.

### Step 2: Serve a Model

In **Terminal 1**, start serving a model. We will keep this intentionally simple:

```bash
vllm serve "RedHatAI/Meta-Llama-3.1-8B-Instruct-quantized.w4a16" \
  --gpu-memory-utilization 0.8
```

For this guide, the only flag we really need to explain is:

- `--gpu-memory-utilization 0.8`: lets vLLM reserve about 80% of GPU memory for serving, which is usually a practical starting point for benchmarking

Everything else is optional and depends on the model, the memory budget, and the specific experiment you want to run.

Wait until you see something like this:

```text
(APIServer pid=92) INFO:     Waiting for application startup.
(APIServer pid=92) INFO:     Application startup complete.
```

Then leave this terminal running.

### Step 3: Warm Up the Model

In **Terminal 2**, run one warm-up pass first:

```bash
vllm bench serve \
  --dataset-name random \
  --model RedHatAI/Meta-Llama-3.1-8B-Instruct-quantized.w4a16 \
  --num-prompts 50 \
  --percentile-metrics ttft,tpot,itl,e2el \
  --random-input-len 2048 \
  --random-output-len 128 \
  --max-concurrency 1
```

Ignore the results from this run. The point is to warm caches and avoid judging cold-start behavior as if it were steady-state performance.

### Step 4: Run the Actual Benchmark

**Single-user style test (`max-concurrency 1`)**

```bash
vllm bench serve \
  --dataset-name random \
  --model RedHatAI/Meta-Llama-3.1-8B-Instruct-quantized.w4a16 \
  --num-prompts 50 \
  --percentile-metrics ttft,tpot,itl,e2el \
  --random-input-len 2048 \
  --random-output-len 128 \
  --max-concurrency 1
```

**Multi-user style test (`max-concurrency 8`)**

```bash
vllm bench serve \
  --dataset-name random \
  --model RedHatAI/Meta-Llama-3.1-8B-Instruct-quantized.w4a16 \
  --num-prompts 50 \
  --percentile-metrics ttft,tpot,itl,e2el \
  --random-input-len 2048 \
  --random-output-len 128 \
  --max-concurrency 8
```

For **VLMs**, do not use a text-only synthetic dataset if you want meaningful numbers. Use a dataset with images, for example:

```bash
vllm bench serve \
  --dataset-name hf \
  --dataset-path lmarena-ai/vision-arena-bench-v0.1 \
  --hf-split train \
  --model <your_vlm_model> \
  --num-prompts 50 \
  --percentile-metrics ttft,tpot,itl,e2el \
  --hf-output-len 128 \
  --max-concurrency 1
```

### What to Focus On

**Output token throughput**

If you see `44.19 tok/s`, read it as: across the benchmark, the system produced output at an average rate of `44.19` generated tokens per second.

Again, this is not just the generation phase by itself. vLLM is measuring throughput over elapsed serving time, so **TTFT is already part of the story**.

**Mean TTFT**

If this rises too much, the system may feel sluggish even if throughput still looks good. For VLMs, TTFT is often higher because image processing is part of the request.

**Mean ITL**

This tells you how smooth the stream feels after generation begins. Lower is better.

**Mean E2EL**

This is the simplest overall latency number. If someone asks, "How long does a request take?" this is often the first number to show them.

## Concurrency 1 vs. Concurrency 8

This is the main trade-off you should expect:

- Higher concurrency usually increases **overall throughput**
- Higher concurrency also tends to increase **TTFT** and **per-request latency**

That is normal. The system is doing more total work, but each request usually gets a smaller share of the device at any moment.

For that reason:

- Use **concurrency 1** to understand the best-case interactive experience
- Use **higher concurrency** to understand server capacity under load

## Final Notes

Keep the benchmark simple. If you are just starting:

1. pick one model
2. warm it up
3. run concurrency `1`
4. run concurrency `8`
5. compare TTFT, ITL, E2EL, and output token throughput together

That gives you a clean first picture of how the model behaves on your Jetson.
